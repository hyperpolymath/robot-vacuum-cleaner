== Claim 1: Dual Julia and Rust implementations with SLAM

[quote, README]
____
A comprehensive, production-grade robot vacuum cleaner simulation system with
dual Julia and Rust implementations, GraphQL API, SLAM algorithms, and
enterprise CI/CD infrastructure.
____

*How it works.*
The Julia implementation lives in `src/julia/RobotVacuum/src/`.
`robot.jl` defines the mutable `Robot` struct with typed fields for battery
level, cleaning mode, heading, dock position, and an `@enum RobotState`
covering Idle / Cleaning / ReturningToDock / Charging / Error / Stuck.
`slam.jl` implements a particle-filter SLAM loop: `OccupancyGrid` holds a
`Matrix{Float32}` updated by ray-casting from sensor readings, while
`ParticleFilter` maintains a `Vector{Particle}` — each carrying a `Pose` —
so localisation and map-building happen simultaneously during every tick.
`pathplanning.jl` provides five strategies (A*, spiral, zigzag, wall-follow,
random) that consume the occupancy grid and emit waypoints consumed by the
simulation loop in `simulator.jl`.
The Rust variant (`src/rust/`) replicates the same architecture for
compute-intensive benchmarks and supply-chain diversity.

*Honest caveat.*
The README uses the phrase "production-grade"; the codebase has genuine
algorithmic depth (particle filter SLAM, multi-mode coverage planning) but
no hardware-in-the-loop validation path.
The GraphQL API (`src/graphql/`) and Prometheus/Grafana observability stack
are structurally present but runtime integration tests are thin — consult
`TEST-NEEDS.md` for the acknowledged gaps before treating coverage claims as
complete.

== Claim 2: Multiple security and quality scanning layers

[quote, README]
____
Security Scanning: Trivy, GitLeaks, cargo audit, Pkg.audit(), OWASP
Dependency Check. Code Quality: JuliaFormatter, Lint.jl, Rust clippy.
____

*How it works.*
`Justfile` holds the canonical recipe set: `just test`, `just lint`,
`just audit`.
Salt state files under `salt/` configure the offline dev environment so
identical quality gates run without network access.
Container definitions in `docker/` use Chainguard Wolfi base images to reduce
supply-chain surface.
`validate-robot-config.k9.ncl` exercises the K9 Nickel schema layer on the
robot configuration at startup, providing a concrete cross-repo integration
with `tree-sitter-k9`.

*Honest caveat.*
The 70 % coverage floor stated in the README depends on `tests/` exercising
the path-planner and SLAM paths through their edge cases — a reviewer should
run `just test` and inspect the Codecov badge before accepting the threshold.

== File Map

[cols="2,4", options="header"]
|===
| Path | What's There

| `src/julia/RobotVacuum/src/robot.jl`
| Core `Robot` struct, `@enum RobotState`, `@enum CleaningMode`, sensor data fields — the central domain model

| `src/julia/RobotVacuum/src/slam.jl`
| Particle-filter SLAM: `OccupancyGrid` (float matrix), `Particle` (pose + weight), `ParticleFilter` — the algorithmic centrepiece

| `src/julia/RobotVacuum/src/pathplanning.jl`
| Five path-planning strategies; A* occupancy-grid search, spiral/zigzag generators, wall-follower, random walk

| `src/julia/RobotVacuum/src/environment.jl`
| Room model: obstacle placement, grid cell encoding, cliff-zone registration

| `src/julia/RobotVacuum/src/simulator.jl`
| Main simulation loop: ticks state machine, applies sensor reads, dispatches planning calls, tracks `RobotStats`

| `src/julia/RobotVacuum/src/types.jl`
| Shared type aliases: `Position`, `Pose`, `SensorData`, `RobotStats`

| `src/julia/RobotVacuum/src/RobotVacuum.jl`
| Package entry point; re-exports the public API surface

| `src/graphql/`
| Julia GraphQL API layer (HTTP.jl) exposing simulation state over a typed schema

| `flake.nix` / `guix.scm`
